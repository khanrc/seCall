## Change description

### 단계별 접근

1. **is_available 단위 테스트 먼저** — 작고 독립적. P47 Rework 1 의 status 분기 회귀 보호.

2. **create_vector_indexer 의 ollama_cloud arm** — 통합 단위 테스트. mockito + tempdir + config 조작.

3. **fallback 결과 assert** — Cloud indexer 인지 fallback indexer 인지 구분이 어려움. 가장 간단한 방법: api_key 없을 때 `try_ollama_fallback_with_ann` 진입 여부를 로컬 Ollama 미가용 환경에서 `None` 반환으로 간접 검증.

### 구현 제약

- **async tokio runtime** — `#[tokio::test(flavor = "multi_thread", worker_threads = 2)]`.
- **mockito server** — 각 테스트마다 server 생성 (port 충돌 회피).
- **`OllamaEmbedder::with_api_key` 가 pub** — P47 에서 builder 패턴으로 작성됨. 외부 호출 가능.
- **fallback indexer 구분 어려움** — `try_ollama_fallback_with_ann` 가 로컬 ollama 안 떠있으면 `None` 반환하므로, "api_key None → None" 만 회귀로 충분.

## Risks

- **fallback 결과 정확한 assert** — Cloud indexer vs fallback indexer 를 외부에서 구분하기 어려움. `VectorIndexer` 내부 embedder 가 trait object 라 type 검사 못함. 가장 명확한 신호는 "api_key None → None / Cloud unreachable → None (로컬 ollama 도 없으면)". `unwrap_or` 같은 silent fallback 대신 명시 assert 가능 형태 사용.
- **로컬 Ollama 가 떠있는 환경** — `try_ollama_fallback_with_ann` 가 로컬 Ollama 발견 시 `Some(_)` 반환. CI 에선 로컬 ollama 없음 → `None` 안전. 로컬 dev 환경에선 결과 다를 수 있음. 테스트는 "create_vector_indexer 가 panic/exception 없이 호출 완료" 까지만 보장하거나, `try_ollama_fallback_with_ann` 만 별도 mock 으로 격리.
- **is_available 분기 검증** — P47 Rework 1 의 status code check 가 회귀하지 않도록 보호. 만약 다시 `send().is_ok()` 로 돌아가면 401 테스트가 실패해야 함.
- code-review-graph 영향: 테스트만 추가, production 코드 변경 0.
